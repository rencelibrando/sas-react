# Activity Proposal — Document Flow

## Overview

This document describes the end-to-end pipeline for processing on-campus student activity proposals in the EARIST SAS Portal. This is a specialized workflow that extends the core incoming/outgoing document model with a sequential, multi-stage approval pipeline.

> **Relationship to Core Backend Flow:** An activity proposal is still modeled as an `incoming` document per the core backend flow. The pipeline described here governs how that document moves through approval stages before SAS generates its response (an Approval Memorandum via the existing Option C flow).

---

## Actors

| Actor | System Role | Login Required? | How They Interact |
|-------|-------------|-----------------|-------------------|
| Student Organization | `student_org` | Yes | Submits documents, tracks status |
| Institute Student Government (ISG) | `isg` | Yes | Assesses org submissions and forwards to SAS; distributes approved docs to org at the end. ISG may **also submit its own activity proposals**, which trigger the FMS + Procurement branch. |
| SAS Admin | `sas_admin` | Yes | Reviews, generates endorsement letter, manages pipeline |
| VPAA | — | No | Receives tokenized email link; approves or returns |
| Office of the President (OP) | — | No | Receives tokenized email link; approves or returns |
| Financial Management Services (FMS) | — | No | (ISG-submitted only) Reviews fund utilization via tokenized email link |
| Procurement Office | — | No | (ISG-submitted only) Reviews procurement via tokenized email link after FMS clearance |

**Office Profiles (VPAA, OP, FMS, Procurement):** None of these offices have system accounts. Their contact email addresses are stored as office profiles managed by SAS Admin (Account Management → Office Review Profiles). SAS Admin is responsible for keeping these up to date when personnel changes occur.

---

## End-to-End Flow

The pipeline branches based on **who submitted the proposal**. The submitter's role
is captured at submission time (`submitterRole` on the document) and used to drive
stage transitions.

### Org-submitted (CSG / AO and other student orgs)

```
Student Org
    │  Submits: request letter + activity form + requirements
    ▼
ISG
    │  Assesses completeness; forwards to SAS (no document generated by ISG)
    ▼
SAS
    │  Reviews; generates Endorsement Letter (outgoing doc)
    │  Sends tokenized review link to VPAA
    ▼
VPAA
    │  Receives tokenized email link; reviews and approves/returns
    ▼
Office of the President
    │  Receives tokenized email link; reviews and approves/returns
    ▼
SAS
    │  Receives approved documents back; performs final release
    ▼
ISG
    │  Receives released documents; distributes to Student Org
    ▼
Student Org
    └  Receives approved activity proposal
```

### ISG-submitted (FMS + Procurement branch)

ISG also runs activities and submits its own proposals. Because ISG activities
draw on institutional funds, FMS and Procurement must clear the fund utilization
after OP approval. ISG-submitted proposals **skip `isg_endorsement`** (ISG is the
requester, not the gatekeeper) and **skip `isg_distribution`** (no separate org
to distribute to — ISG receives the result directly).

```
ISG (submitter)
    │  Submits: same form as student orgs + Fund Utilization input
    ▼
SAS
    │  Reviews; generates Endorsement Letter
    │  Sends tokenized review link to VPAA
    ▼
VPAA  →  Office of the President
    │
    ▼
FMS
    │  Receives tokenized email link; reviews fund utilization; approves/returns
    ▼
Procurement Office
    │  Receives tokenized email link; reviews procurement; approves/returns
    ▼
SAS
    │  Performs final release — proposal becomes status: "approved"
```

## Pipeline Stages

The activity proposal document carries a `pipeline` object that tracks progression through six named stages.

### Stage Definitions

| Stage Key | Responsible Actor | Action | Org-Submitted? | ISG-Submitted? |
|-----------|-------------------|--------|----------------|----------------|
| `isg_endorsement` | ISG | Assesses submission for completeness; forwards to SAS | ✓ | — (skipped) |
| `sas_review` | SAS Admin | Reviews documents; generates Endorsement Letter | ✓ | ✓ (entry stage) |
| `vpaa_review` | VPAA (tokenized) | Reviews via email link; approves or returns | ✓ | ✓ |
| `op_approval` | OP (tokenized) | Reviews via email link; approves or returns | ✓ | ✓ |
| `fms_review` | FMS (tokenized) | Reviews fund utilization via email link; approves or returns | — | ✓ |
| `procurement_review` | Procurement (tokenized) | Reviews procurement via email link; approves or returns | — | ✓ |
| `sas_release` | SAS Admin | Final release | ✓ (→ ISG distribution) | ✓ (→ approved) |
| `isg_distribution` | ISG | Distributes released documents to Student Org | ✓ | — (skipped) |

> **ISG does not generate any documents.** The Endorsement Letter is generated by SAS Admin at the `sas_review` stage using the core Option C outgoing document flow.

### Stage Transitions

```
[isg_endorsement]
      │ ISG assesses and forwards
      ▼
[sas_review]
      │ SAS reviews + generates Endorsement Letter + sends to VPAA
      ▼
[vpaa_review]
      │ VPAA approves via tokenized link
      ▼
[op_approval]
      │ OP approves via tokenized link
      ▼
[sas_release]
      │ SAS performs final release
      ▼
[isg_distribution]
      │ ISG distributes to Student Org
      ▼
[completed — pipeline.currentStage = null, status = "approved"]
```

At any stage, an actor may **return** the document with remarks. The pipeline regresses to the appropriate prior stage for revision.

## Tokenized Review Links (VPAA & OP)

VPAA and OP do not log into the system. Instead, when the pipeline reaches their stage, the system:

1. Generates a **unique, single-use token** tied to the document and the specific stage.
2. Sends an **email** to the office's registered email address containing a review link:
   ```
   https://yourapp.com/review?token=<token>
   ```
3. The link opens a read-only review page showing:
   - The activity proposal and all attached documents
   - The SAS-generated Endorsement Letter
   - An **Approve** button and a **Return** button (with remarks field)
4. On action, the token is **consumed** (one-time use) and the pipeline advances or regresses accordingly.

### Token Behavior

| Concern | Behavior |
|---------|----------|
| Expiry | Token expires after a configurable period (e.g., 7 days) |
| Expired token | Link shows expiry notice; SAS Admin can resend from the pipeline dashboard |
| Consumed token | Link shows "This action has already been completed" |
| Security note | The link is effectively a credential — it is tied to one document and one stage only |

### SAS Visibility

SAS Admin can see:
- When the token email was sent (`tokenSentAt`)
- Whether the token has been consumed (`completedAt`)
- How many days have elapsed since sending
- A **Resend** button to regenerate and resend if the token has expired or gone missing

## Data Model

The activity proposal document extends the core document schema with a `pipeline` field:

```javascript
{
  documentId: "abc123",
  documentType: "activity_proposal",
  direction: "incoming",
  organizationId: "org456",
  status: "pending",            // remains "pending" throughout the pipeline;
                                // becomes "approved" when isg_distribution completes

  pipeline: {
    currentStage: "vpaa_review",  // null when pipeline is complete
    stages: [
      {
        stage: "isg_endorsement",
        action: "forwarded",
        completedBy: "isg_user_id",
        completedAt: timestamp,
        remarks: "Forwarded to SAS."
      },
      {
        stage: "sas_review",
        action: "forwarded",
        completedBy: "sas_admin_id",
        completedAt: timestamp,
        generatedDocId: "endorsement_letter_doc_id"  // linked outgoing doc created by SAS
      },
      {
        stage: "vpaa_review",
        token: "abc123xyz",
        tokenSentAt: timestamp,
        tokenExpiresAt: timestamp,
        completedAt: null,        // null = still pending
        completedBy: "vpaa@earist.edu.ph",
        remarks: null
      },
      // op_approval, sas_release, isg_distribution not yet initiated
    ]
  }
}
```

### Key Design Notes

- **ISG generates no documents.** The `isg_endorsement` stage appends a pipeline entry with `action: "forwarded"` and advances `currentStage` to `sas_review`. No files are added by ISG.
- **Endorsement Letter is SAS's responsibility.** Created at `sas_review` via the core Option C outgoing document flow. The `generatedDocId` in the stage entry links back to that outgoing document.
- **`status` field stays `"pending"`** throughout the pipeline. `pipeline.currentStage` is the source of truth for where the document is. The terminal transition depends on the submitter:
  - **Org-submitted:** on `isg_distribution` completion, `status` → `"approved"`, `pipeline.currentStage` → `null`.
  - **ISG-submitted:** on `sas_release` completion, `status` → `"approved"`, `pipeline.currentStage` → `null` (no `isg_distribution` stage).
- **`submitterRole` is recorded on the document at submission** — `"ISG"` triggers the FMS + Procurement branch; any other value uses the org-submitted flow. The post-`op_approval` next-stage selection in `server.js` reads this field.
- **Fund Utilization** is collected as an uploaded document (requirement key `fund_utilization`) only when an ISG user submits the proposal. It appears alongside the other requirement files and is reviewed by FMS / Procurement via their tokenized review pages.

## Role-Specific Views

### Student Organization
- Sees current pipeline stage (read-only, human-readable label)
- Cannot see internal remarks between offices
- Notified when the document is fully approved and released by ISG

### ISG
- **Entry (`isg_endorsement`):** Sees pending submissions from all student organizations; assesses completeness; forwards to SAS or returns with remarks
- **Exit (`isg_distribution`):** Sees documents released by SAS; marks as distributed once handed to Student Org

### SAS Admin
- Full pipeline dashboard: sees all documents and their current stage
- At `sas_review`: triggers Option C to generate Endorsement Letter; advances pipeline which sends tokenized email to VPAA
- At `sas_release`: performs final release after OP approval
- Can resend tokenized emails for VPAA and OP stages
- Can view full stage history with timestamps

### VPAA / OP
- No login required
- Accesses review page via tokenized email link
- Sees document + endorsement letter; clicks Approve or Return

## Required Documents at Submission

Per the official SAS checklist, the Student Org must attach all of the following before ISG can forward:

1. Request Letter (addressed to ISG President through College Student Government)
2. Request Letter (addressed to Institute President through VPAA, SAS Director, Program Head)
3. Student Activity Proposal Form (EARIST-QSF-SAS-006)
4. Budgetary Allocation and Venue Reservation
5. Profile of Speakers/Facilitators (for seminars/workshops)
6. Program / Event Flow
7. Resolution on fee collection + minutes of meeting *(if collecting registration fees)*
8. Other requirements as specified by SAS

> **Completeness Gate:** Processing begins only upon receipt of complete requirements. The system enforces this at the ISG stage — ISG cannot forward an incomplete submission.

## Edge Cases

| Scenario | Behavior |
|----------|----------|
| Document returned at VPAA | Pipeline regresses to `sas_review`; SAS Admin notified with VPAA's remarks |
| Document returned at OP | Pipeline regresses to `sas_review`; SAS Admin notified with OP's remarks |
| Token expired before VPAA acts | SAS Admin sees "Token expired X days ago"; resends with new token |
| ISG forwards incomplete documents | Completeness gate blocks forwarding; Student Org must revise and resubmit |
| SAS Admin tries to skip a stage | Not allowed; stage transitions are strictly sequential and enforced server-side |

---

## Summary

The activity proposal pipeline is the most complex workflow in the EARIST SAS Portal. It introduces:

- **A sequential 6-stage pipeline** that extends the core incoming document model
- **Tokenized review links** for VPAA and OP, eliminating the need for external accounts while maintaining an audit trail
- **A completeness gate** at ISG assessment to prevent incomplete submissions from entering the pipeline
- **SAS as the central coordinator** — consistent with the hub model — with full visibility and control at every stage
- **ISG as both entry gatekeeper and exit distributor**, with distinct actions at each end of the pipeline

The SAS-generated Endorsement Letter at the `sas_review` stage is produced via the existing Option C mechanism from the core backend flow, maintaining consistency across all document types.
